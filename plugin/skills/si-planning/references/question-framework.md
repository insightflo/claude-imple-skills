# SI Planning — 질문 프레임워크

> **목적**: 소크라테스식 대화로 고객의 암묵지를 끌어내어 구조화된 요구사항으로 변환한다.
> Ambiguity Score 0.3 이하가 될 때까지 질문을 반복한다.

---

## 1. 질문 카테고리 체계

총 5개 카테고리. 각 카테고리는 독립적이지 않으며 답변에서 다음 카테고리의 질문을 유도한다.

```
Category 1: 프로젝트 개요 (Project Overview)
Category 2: As-Is / To-Be 분석
Category 3: 기능 요구사항 (Functional Requirements)
Category 4: 비기능 요구사항 (Non-Functional Requirements)
Category 5: 이해관계자 분석 (Stakeholder Analysis)
```

---

## Category 1: 프로젝트 개요

### 기본 질문 세트

```
[C1-Q01] 이 프로젝트의 공식 명칭과 발주처는 어디인가요?
[C1-Q02] 이 시스템을 만들게 된 가장 직접적인 계기는 무엇인가요?
[C1-Q03] 프로젝트 완료 목표 일정은 언제인가요?
[C1-Q04] 예산 규모는 대략 어느 정도인가요? (범위로 답해도 됩니다)
[C1-Q05] 투입 인력은 어느 정도 예상하시나요? (개발팀 기준)
[C1-Q06] 이 프로젝트의 성공을 어떻게 정의하시나요?
```

### 심화 질문 (C1-Q01~06 답변 후 불명확 시)

```
[C1-Q07] 이 프로젝트가 무산된다면 발주처에서 가장 큰 타격을 받는 부서는 어디인가요?
[C1-Q08] 프로젝트 완료 이후 시스템 운영/유지보수 주체는 누구인가요?
[C1-Q09] 비슷한 프로젝트를 이전에 시도한 적이 있나요? 있다면 왜 실패했나요?
[C1-Q10] 이 프로젝트의 최우선 제약 조건은 무엇인가요? (일정 vs 예산 vs 품질 중 하나 선택)
```

### AskUserQuestion 통합 예시

```python
# Phase 0 → Phase 1 전환 시 첫 질문
AskUserQuestion(
  question="""SI 프로젝트 요구사항 수집을 시작하겠습니다.

먼저 프로젝트 기본 정보를 알려주세요:

1. 프로젝트명과 발주처
2. 이 시스템을 만들게 된 배경 (한 문장으로)
3. 목표 완료 일정
4. 이 시스템이 없으면 어떤 일이 벌어지나요?

(모르는 항목은 '미정' 또는 '모름'으로 적어주세요)""",
  options=None  # 자유 텍스트 답변
)
```

## Category 2: As-Is / To-Be 분석

> 현재 상태(As-Is)를 깊이 이해해야 미래 상태(To-Be)가 구체화된다.
> 고객은 종종 To-Be를 말하려 하지만, As-Is를 먼저 탐색해야 숨은 요구사항이 드러난다.

### As-Is 탐색 질문

```
[C2-Q01] 현재 이 업무를 어떻게 처리하고 있나요? (시스템? 엑셀? 수작업?)
[C2-Q02] 아침에 출근하면 이 업무와 관련해서 가장 먼저 하는 일은 무엇인가요?
[C2-Q03] 하루에 이 업무 처리에 얼마나 시간을 쓰나요?
[C2-Q04] 지금 가장 불편한 점 3가지를 꼽아주세요.
[C2-Q05] 지금 쓰고 있는 시스템/도구/문서 양식은 무엇인가요?
[C2-Q06] 이 업무에서 '실수'가 가장 자주 발생하는 단계는 어디인가요?
[C2-Q07] 현재 시스템에서 데이터를 어디에 어떻게 보관하나요?
[C2-Q08] 현재 프로세스에서 사람이 직접 확인해야 하는 단계는 어디인가요? (자동화 후보)
```

### To-Be 도출 질문

```
[C2-Q09] 이 시스템이 완성되면, C2-Q04에서 말한 불편함이 어떻게 해결되나요?
[C2-Q10] 6개월 후 이 시스템을 쓰는 담당자가 가장 행복해하는 기능 하나는?
[C2-Q11] 이 시스템 없이는 절대 안 되는 핵심 기능은 무엇인가요?
[C2-Q12] 현재 하루에 N시간 걸리는 업무가, 이 시스템으로 얼마나 줄어야 성공인가요?
[C2-Q13] 이 시스템에서 '인간의 판단'이 반드시 필요한 단계는 어디인가요?
```

### As-Is/To-Be 갭 자동 분석

답변 수집 후 다음 구조로 분석:

```markdown
## As-Is / To-Be Gap 분석

| 항목 | As-Is | To-Be | Gap | 우선순위 |
|------|-------|-------|-----|---------|
| 데이터 입력 | 수작업 엑셀 | 시스템 자동화 | 입력 자동화 | Must |
| 승인 프로세스 | 종이 결재 | 전자결재 | 워크플로우 | Must |
| 보고서 생성 | 수작업 취합 | 자동 생성 | 리포팅 | Should |
```

## Category 3: 기능 요구사항

### 정방향 질문 (Top-Down)

```
[C3-Q01] 이 시스템의 핵심 기능을 업무 흐름 순서대로 말씀해 주세요.
[C3-Q02] 사용자가 매일 사용할 기능 3가지는 무엇인가요?
[C3-Q03] 월말/연말 등 특정 시기에만 필요한 기능이 있나요?
[C3-Q04] 관리자와 일반 사용자의 기능 차이는 어떻게 되나요?
[C3-Q05] 이 기능이 완료되었을 때 시스템이 어떤 결과물을 내놓아야 하나요?
```

### 역방향 질문 (Anti-Requirements)

> 금지 사항을 정의하면 숨은 기능 요구사항이 드러난다.

```
[C3-Q06] 이 시스템에서 절대 일어나면 안 되는 상황은 무엇인가요?
         (예: "데이터 삭제가 돌이킬 수 없어서는 안 된다" → 소프트 삭제 요구사항 도출)
[C3-Q07] 이전에 쓰던 시스템에서 가장 큰 사고/장애가 무엇이었나요?
         (예: "이중 결제" → 멱등성, 잠금 요구사항 도출)
[C3-Q08] 특정 사용자가 절대 볼 수 없어야 하는 데이터는 무엇인가요?
         (예: "인사 데이터는 인사팀만" → 세밀한 권한 요구사항 도출)
[C3-Q09] 이 기능이 잘못 동작했을 때 법적/재정적 책임이 발생하는 경우가 있나요?
```

### 숨은 요구사항 탐색 질문

```
[C3-Q10] 이 데이터를 누가 또 보나요? (다운스트림 이해관계자 발굴)
[C3-Q11] 이 기능이 장애가 나면 어떤 부서/업무가 멈추나요? (크리티컬 패스 분석)
[C3-Q12] 데이터가 틀리면 누가 책임지나요? (데이터 거버넌스 요구사항)
[C3-Q13] 이 데이터를 외부 기관에 제출해야 하는 경우가 있나요? (연동/변환 요구사항)
[C3-Q14] 지금 이 업무를 담당자가 갑자기 사라지면 누가 대신할 수 있나요?
         (인수인계 → 시스템 매뉴얼/도움말 기능 요구사항)
[C3-Q15] 이 시스템 데이터를 몇 년간 보관해야 하나요? (보존 정책, 아카이브 요구사항)
```

### 재난 시나리오 질문 (SI 필수)

```
[C3-Q16] 시스템이 완전히 다운되면 업무가 얼마나 버틸 수 있나요?
[C3-Q17] 네트워크가 단절되어도 현장에서 업무를 계속해야 하는 경우가 있나요?
         (오프라인 모드 요구사항)
[C3-Q18] 서버가 폭발해도 데이터를 복구할 수 있어야 하나요? 어느 시점까지?
         (RPO/RTO 요구사항)
[C3-Q19] 랜섬웨어 공격을 받으면 어떻게 되나요? 백업 정책은?
```

### 유사 시스템 레퍼런스 질문

```
[C3-Q20] 비슷한 시스템을 사용해 본 적 있나요? 어떤 시스템이었나요?
[C3-Q21] 그 시스템에서 가장 마음에 들었던 기능 2가지는?
[C3-Q22] 그 시스템에서 반드시 피하고 싶은 점은?
[C3-Q23] 경쟁사나 타 기관에서 이미 잘 쓰고 있는 시스템이 있나요?
```

## Category 4: 비기능 요구사항 (ISO 25010 기반)

### 성능 효율성 (Performance Efficiency)

```
[C4-Q01] 동시 접속자 수는 최대 몇 명 정도인가요? (피크 타임 기준)
[C4-Q02] 가장 자주 쓰는 화면/기능의 응답 속도는 몇 초 이내여야 하나요?
[C4-Q03] 대용량 데이터를 한 번에 처리해야 하는 경우가 있나요?
         (예: 10만 건 엑셀 업로드, 배치 처리)
[C4-Q04] 데이터 증가 속도는 어느 정도인가요? (연간 몇 건/GB?)
[C4-Q05] 피크 타임은 언제인가요? (월말 정산, 연말 결산 등)
```

### 신뢰성 (Reliability)

```
[C4-Q06] 시스템 장애 시 허용 가능한 최대 중단 시간은? (분/시간 단위)
[C4-Q07] 1년 중 시스템이 반드시 가동되어야 하는 기간은?
         (예: "연말 2주는 절대 다운되면 안 됨")
[C4-Q08] 장애 발생 시 데이터 손실 허용 범위는?
         (RPO: 최대 N분/시간 전 데이터까지 복구 가능해야)
[C4-Q09] 장애 복구 목표 시간은? (RTO: N시간 이내 복구)
[C4-Q10] 계획된 유지보수를 위한 다운타임이 허용되나요? 언제가 좋은가요?
```

### 보안성 (Security)

```
[C4-Q11] 이 시스템에서 가장 민감한 데이터는 무엇인가요?
[C4-Q12] 외부 인터넷에서 접근해야 하나요, 아니면 내부망만인가요?
[C4-Q13] 개인정보보호법/특정 보안 규정을 준수해야 하나요?
[C4-Q14] 보안 감사/인증이 필요한 경우가 있나요? (ISMS-P, ISO 27001 등)
[C4-Q15] 내부자 위협(악의적 내부자)에 대한 대비가 필요한가요?
         → 접근 로그, 이상 탐지 요구사항 도출
```

### 유지보수성 (Maintainability)

```
[C4-Q16] 시스템 오픈 후 기능을 자주 변경해야 하는 부분이 있나요?
[C4-Q17] 발주처 내부 IT 인력이 직접 운영/수정하는 경우가 있나요?
[C4-Q18] 소스 코드 공개 또는 기술 이전이 필요한가요?
```

### 사용성 (Usability)

```
[C4-Q19] 주 사용자의 IT 친숙도는 어느 정도인가요?
         (고령자, 현장 직원, IT 전문가 등)
[C4-Q20] 접근성 기준을 충족해야 하나요? (장애인 접근성, 웹접근성 AA 등)
[C4-Q21] 모바일 환경 지원이 필요한가요? (반응형 or 네이티브 앱)
[C4-Q22] 다국어 지원이 필요한가요?
```

### 이식성/호환성 (Portability / Compatibility)

```
[C4-Q23] 기존 시스템과 연동해야 하는 것이 있나요? (ERP, 레거시 등)
[C4-Q24] 지원해야 하는 브라우저/OS 환경 제약이 있나요?
[C4-Q25] 클라우드 vs 온프레미스 정책이 이미 결정되어 있나요?
```

## Category 5: 이해관계자 분석

### 직접 이해관계자 파악

```
[C5-Q01] 이 시스템을 직접 사용하는 사람들을 역할별로 나눠주세요.
[C5-Q02] 각 역할별 사용 빈도와 주요 업무는 무엇인가요?
[C5-Q03] 이 시스템 도입으로 가장 이익을 얻는 사람/부서는 어디인가요?
[C5-Q04] 이 시스템 도입으로 업무가 줄거나 없어지는 사람이 있나요?
         (저항 이해관계자 → 변화 관리 요구사항)
```

### 숨은 이해관계자 탐색

```
[C5-Q05] 이 시스템의 결과물(보고서, 데이터)을 보고받는 상위 관리자/기관은?
[C5-Q06] 이 시스템에 데이터를 '입력하는 사람'과 '결과를 보는 사람'이 다른가요?
[C5-Q07] 외부 기관(정부, 협력사, 감독기관)과 데이터를 주고받나요?
[C5-Q08] 이 시스템 감사/점검을 수행하는 내부/외부 기관이 있나요?
[C5-Q09] IT 인프라 운영팀은 별도로 있나요? 그들의 요건은?
[C5-Q10] 이 시스템을 직접 사용하지 않지만 영향을 받는 고객/시민이 있나요?
```

### 이해관계자 관심사 매핑

```
[C5-Q11] 각 이해관계자가 이 시스템에서 가장 중요하게 생각하는 것은?
         (경영진: 비용절감/리포팅 / 실무자: 편의성/속도 / IT팀: 안정성/유지보수성)
[C5-Q12] 이해관계자 간에 요구사항이 충돌하는 부분이 있나요?
         (예: "경영진은 모든 데이터 실시간 공개" vs "현장팀은 입력 부담 최소화")
[C5-Q13] 최종 의사결정권자는 누구인가요? (요구사항 충돌 시 tiebreaker)
```

## 동적 질문 생성 규칙

> 고객의 답변 패턴에 따라 후속 질문을 다르게 생성한다.

### 모호한 답변 대응 (Vague Answer Pattern)

```
트리거: "빠르게", "편하게", "좋게", "보통", "적당히" 등 정량화 불가 표현

후속 질문 전략:
  "빠르게"   → "현재 몇 초 걸리고, 몇 초면 빠른 건가요?"
  "많은 사용자" → "구체적으로 몇 명인가요? 동시에 몇 명이 쓰나요?"
  "중요한 데이터" → "이 데이터가 잘못되면 어떤 일이 벌어지나요?"
  "일반적인 보안" → "어떤 보안 인증이 필요하신가요? ISMS-P나 CC인증 등이요?"
  "나중에 결정" → "이것이 정해지지 않으면 어떤 기능을 설계할 수 없게 되나요?"

정량화 요청 패턴:
  - "N% 향상"이라면 → "현재 기준선(Baseline)이 무엇인가요?"
  - "빠른 응답"이라면 → "몇 초 이내를 원하시나요? 3초? 1초?"
  - "안정적으로"라면 → "1년에 몇 시간 다운되는 게 허용되나요?"
```

### 과잉 답변 대응 (Over-Specification Pattern)

```
트리거: 기술 구현 세부사항을 요구사항으로 언급하는 경우
  예: "React로 만들어주세요", "MySQL 써야 합니다", "마이크로서비스로 해주세요"

후속 질문 전략:
  기술 스택 언급 → "왜 그 기술이어야 하나요? 기존 시스템과의 연동 때문인가요,
                    아니면 내부 IT 정책 때문인가요?"
  구현 방식 지정 → "이 기능의 목적은 무엇인가요? 그 목적을 달성하는 다른 방법도
                    검토해봐도 될까요?"

정책: 기술 구현 세부사항은 요구사항이 아닌 '제약 조건'으로 분류 후
      추후 아키텍처 설계 단계로 이관
```

### 회피 답변 대응 (Avoidance Pattern)

```
트리거: "모르겠어요", "나중에 봐요", "알아서 해주세요", "다 필요해요"

후속 질문 전략:
  "모르겠어요" → "그 결정을 내릴 수 있는 다른 분이 있나요?
                  아니면 이 부분을 저희가 기본값으로 제안드릴까요?"

  "다 필요해요" → "만약 예산이 지금의 절반이라면 포기할 수 있는 기능은?"
                  (Forced Ranking 기법)

  "알아서 해주세요" → "이 결정에서 잘못되면 가장 큰 문제가 뭔가요?
                       그 리스크를 저희가 지기를 원하시나요?"
```

## Ambiguity Score 측정 시스템

### 6개 측정 차원

```
[Dimension 1] 문제 정의 명확성 (Problem Definition)
  측정 기준:
    1.0 = As-Is/To-Be 미정의, 왜 필요한지 불명
    0.8 = As-Is만 있고 To-Be 없음
    0.5 = To-Be 있지만 정량 목표 없음
    0.3 = As-Is/To-Be 있고 성공 기준 일부 정의
    0.1 = As-Is/To-Be 명확, 정량 성공 기준 완비
    0.0 = 이전 유사 프로젝트 비교까지 완료

[Dimension 2] 대상 사용자 명확성 (User Clarity)
  측정 기준:
    1.0 = "직원들이 씁니다" 수준
    0.8 = 역할 구분은 있지만 각 역할의 업무 불명
    0.5 = 역할별 주요 업무 파악, 이해관계자 일부 미발굴
    0.3 = 직접/간접 이해관계자 대부분 식별
    0.1 = 이해관계자별 요구사항 상충 관계까지 파악
    0.0 = 이해관계자 우선순위 및 tiebreaker 확정

[Dimension 3] 핵심 기능 구체성 (Feature Clarity)
  측정 기준:
    1.0 = "관리 기능이 필요해요" 수준
    0.8 = 기능 목록 존재하나 각 기능의 입출력 불명
    0.5 = 기능별 입출력 있으나 엣지 케이스 미정의
    0.3 = 정상 흐름 + 주요 예외 흐름 정의
    0.1 = Anti-requirements 및 재난 시나리오까지 정의
    0.0 = 기능 간 종속성 그래프 완성

[Dimension 4] 비기능 요건 정량화 (NFR Quantification)
  측정 기준:
    1.0 = NFR 미정의
    0.8 = 일부 NFR 정의, 대부분 정성적
    0.5 = 주요 NFR 정의, 성능/가용성 수치 일부
    0.3 = ISO 25010 6개 이상 특성 정의
    0.1 = 모든 주요 NFR 정량화 (SLA 포함)
    0.0 = 도메인 규정 준수 NFR까지 완비

[Dimension 5] 범위 경계 명확성 (Scope Boundary)
  측정 기준:
    1.0 = 범위 경계 없음
    0.8 = 포함 항목만 있고 제외 항목 불명
    0.5 = 주요 포함/제외 있으나 경계 사례 모호
    0.3 = 포함/제외 명확, Won't 목록 작성됨
    0.1 = 연동 시스템 경계까지 명확히 정의
    0.0 = Phase 구분(1차/2차) 및 마일스톤 확정

[Dimension 6] 성공 기준 정의 (Success Criteria)
  측정 기준:
    1.0 = "잘 만들어주세요" 수준
    0.8 = 일반적 성공 기준 ("편리하게", "빠르게")
    0.5 = 일부 정량 기준 (응답 속도, 오류율)
    0.3 = 검수 조건 초안 있음
    0.1 = 정량적 검수 기준서 완비
    0.0 = 인수 테스트 시나리오까지 확정
```

### Score 계산 및 진입 조건

```
Total Ambiguity Score = (D1 + D2 + D3 + D4 + D5 + D6) / 6

진입 조건:
  Score > 0.6  → 계속 질문 (주요 영역 대부분 불명확)
  Score 0.3~0.6 → 불명확 차원 집중 질문
  Score < 0.3  → Phase 2 진입 가능

권고 임계값:
  Must have: Score <= 0.3
  Best practice: Score <= 0.2

단, 특정 차원이 0.7 이상이면 전체 Score와 무관하게
해당 차원 집중 질문 후 Phase 2 진입
```

## 공공기관 SI 프로젝트 질문 흐름 예시

> 시나리오: 지방자치단체 민원 처리 시스템 구축

### Round 1 — 첫 AskUserQuestion

```
[AI]: 안녕하세요. 민원 처리 시스템 구축 프로젝트 요구사항 수집을 시작하겠습니다.

현재 민원 업무가 어떻게 이루어지고 있는지 알려주세요:

1. 현재 민원을 어떻게 접수받고 있나요? (방문, 전화, 팩스, 이메일, 온라인?)
2. 접수된 민원을 처리하는 데 평균 며칠 걸리나요?
3. 담당자 배정은 어떻게 하나요?
4. 처리 완료를 민원인에게 어떻게 알리나요?

[담당자]: 지금은 방문이랑 전화가 70%고, 나머지는 이메일로 옵니다.
처리 기간은 법정 기한이 7일인데 실제로는 10일 이상 걸리는 경우도 있어요.
담당자 배정은 수기로 하고, 카카오톡이나 문자로 알려주고 있어요.
```

### Round 2 — 갭 탐지 후 심화 질문

```
[AI]: 법정 기한(7일)을 초과하는 경우가 있다고 하셨는데,

1. 초과하는 주된 이유가 무엇인가요? (담당자 부재? 정보 조회 지연? 내부 결재?)
2. 법정 기한 초과 시 어떤 패널티나 민원이 발생하나요?
3. 특정 민원 유형이 유독 오래 걸리나요?

[담당자]: 주로 내부 결재가 오래 걸려요. 과장님이 출장 중이면 일주일씩 밀리기도 하고요.
건축 인허가 관련 민원은 타 부서 협의가 필요해서 특히 오래 걸립니다.
```

### Round 3 — 숨은 요구사항 발굴

```
[AI]: 타 부서 협의가 필요한 경우가 있다고 하셨는데,

1. 어떤 부서들과 협의가 필요한가요? (부서 목록)
2. 그 협의 과정이 현재는 어떻게 진행되나요? (이메일? 전화? 직접 방문?)
3. 협의 결과를 기록/관리하는 방법이 있나요?
4. 감사원이나 상위 기관에 민원 처리 현황을 보고해야 하는 경우가 있나요?

→ 여기서 발굴된 숨은 요구사항:
   - 부서 간 협의 워크플로우 (예상 못했던 기능)
   - 결재 권한 위임 기능 (대결/전결)
   - 감사 로그 및 행정 감사 대응 리포팅
   - 타 부서 시스템과의 연동 (GIS, 건축행정시스템 등)
```

## 질문 종료 조건

### 종료 조건 (AND 조건)

```
1. Ambiguity Score <= 0.3 (전체)
2. 모든 차원 Score <= 0.5 (단일 차원 블로커 없음)
3. 도메인 필수 체크리스트 질문 완료
4. 이해관계자 목록에 "미확인" 항목 없음
5. Must 요구사항 목록이 최소 5건 이상
6. 성공 기준이 최소 1개 이상 정량화됨
```

### 조기 종료 허용 조건

```
사용자가 명시적으로 "이걸로 충분합니다" 또는 "시작합시다"를 요청 AND
  - Ambiguity Score <= 0.5
  - 최소 10개 이상의 요구사항 수집됨

조기 종료 시 경고 메시지:
  "현재 Ambiguity Score가 0.XX로 일부 불명확한 부분이 있습니다.
   다음 항목은 설계/개발 단계에서 재확인이 필요합니다:
   - [불명확 항목 목록]
   진행하시겠습니까?"
```

### 강제 종료 금지 조건

```
다음 상태에서는 Phase 1을 종료할 수 없음:
  - Must 요구사항이 0건
  - 이해관계자가 1명도 식별되지 않음
  - 성공 기준이 전혀 정의되지 않음 (D6 = 1.0)
  - 도메인 필수 규정 체크리스트 미완료
```
